Wires and Ports

Understanding how data flows through wires is essential to building effective patches. Every port has a type and a direction, and every wire carries exactly one type of data from an output to an input.

Port Types

TypeScopeWire ColorWhat It Carries
fieldFloatPer-pin (GPU)GreyA single float computed for every point in the cloud simultaneously. 200,000 independent values per frame. Used for depth nearness, size, noise, masks.
fieldVec3Per-pin (GPU)GreenA 3D vector per point. Carries position offsets (xyz displacement from pin home), rotation (euler turns), stretch (per-axis scale).
fieldColorPer-pin (GPU)PinkRGBA color per point. Routed through palette lookups, duotone, contrast, fog — then into Output.color.
signalPer-frame (CPU)WhiteA single float value computed once per frame on the CPU. Used for LFOs, audio volume, hand position, MIDI CC. Broadcasts to every pin when wired into a fieldFloat input.
triggerPer-frame (CPU)OrangeAn event pulse — 1 on the frame it fires, 0 otherwise. Beat detection, gesture recognition, threshold crossing. Drives Shockwave fire, Freeze hold, Envelope input.
vec3Per-frame (CPU)BlueA 3D vector computed once per frame. Used for hand position, head pose.
colorPer-frame (CPU)PinkRGBA color computed once per frame.

Input vs Output Squares

On every node card, ports are shown as small squares:

SquarePositionMeaning
OutputRight side of nodeData this node produces. Drag FROM here to start a wire. One output can feed many inputs.
InputLeft side of nodeData this node consumes. Drag TO here to complete a wire. Each input accepts exactly ONE wire. If unwired, the port uses its default value (shown in the port table).

fieldFloatfieldVec3Splat. Each pin's float becomes a vec3 with that value in all channels.

The depth vs z distinction

The Depth node produces two fundamentally different outputs that are easily confused:

OutputTypeWhat It IsUse For
depthfieldFloatNearness. A normalized 0–1 modulation signal. Computed as (far − raw_metres) / (far − near). 1 = at NEAR, 0 = at FAR or beyond. Holes read 0. No scale or sign — pure "how close is this pixel."Palette.t, Size modulation, Shape-by-Depth, Gate thresholds, mask inputs
zfieldFloatGeometry. The point's actual forward displacement in view units. Computed as (FOCUS − raw_metres) × SEPARATION. Signed — negative behind the wall for things farther than FOCUS. Metrically scaled so the cloud is 1:1 with reality.Output.z pin placement, Echo/Lag/Trail/Spring/Velocity motion processing

Concrete example: A point at 4 metres with FAR = 2.5 reads depth = 0 (clamped — "far away, color it cold") but z = −7.5 view units (a real position behind the wall). Wire depth into Output.z by mistake and the entire scene compresses into a flat 0–1 push. Wire z into Palette.t and colors won't map sensibly because z is signed and unbounded.

Rule: depth feeds effects. z builds the cloud. Never cross them.

Depth

SOURCE GPU node. The primary sensor — reads the depth camera and produces three outputs that form the foundation of every patch. This is the only node that talks to the hardware. Without it (or Vision Model), there are no points.

How It Works

The Depth node sits at the boundary between the physical world and the GPU. Every frame, the depth sensor (LiDAR, TrueDepth, or RGB luma) produces a texture where each pixel holds a distance in metres. The node reads that texture and computes three things per pin:

1. Nearness (the depth output): Remaps raw metres into a clean 0–1 signal. The formula is (far − raw) / (far − near). A point at the NEAR distance reads 1.0. A point at or beyond FAR reads 0.0. Sensor holes also read 0. This is your modulation signal — wire it into Palette.t to color by distance, into Size to scale by distance, into a Gate's threshold to trigger on proximity.

2. Position offset (the position output): A 3D vector per pin in view space. In METRIC mode this uses the real camera intrinsics — objects keep true size and shrink as they move away, exactly like your eyes see them. In FREE mode it's a simpler fan projection controlled by GAIN. This feeds Pinout.position, and from there Output.position.

3. Z push (the z output): A signed scalar — the point's forward displacement from the wall. Computed as (FOCUS − raw) × SEPARATION. Positive = in front of the wall (closer than FOCUS). Negative = behind the wall (farther than FOCUS). This is the geometry signal — feed it into Output.z, or process it through Lag/Trail/Echo/Spring/Velocity before it reaches the renderer.

Parameters

ParamRangeDefaultWhat It Does
near0.05 – 50.1Distance in metres that maps to depth=1. TrueDepth reads down to ~10cm — set higher to avoid the "flat face" plateau when too close.
far0.2 – 82.5Distance in metres that maps to depth=0. Everything at or beyond this reads as "far away."
invertboolfalseFlips the nearness signal: far→1, near→0. Useful when you want background emphasis instead of foreground.
modefree / metricmetricMETRIC: Uses real camera intrinsics (fx,fy,cx,cy). Objects keep their true relative size at any distance — 1:1 with physical reality. FREE: Simpler fan projection controlled by GAIN. No intrinsics needed.
separation0 – 42.5How many view units one real-world metre maps to. At 2.5, a point 1m in front of FOCUS appears 2.5 units forward. Higher = more exaggerated depth.
focus0.3 – 31.0The depth (in metres) that sits exactly at the Z-wall — no push, no pull. Things closer than FOCUS push forward. Things farther push backward.
gain0 – 32.5Z zoom strength in FREE mode. Higher = deeper push for the same nearness.
armsboolfalsePulls every point back to its Z-origin pin. Creates the "rod" effect — each pin has a visible stem connecting the point to the wall. Style the stems with the Stem node.
edgeCull0 – 0.30.06Built-in silhouette flying-pixel rejection. Depth edges are noisy — points near object boundaries jitter. This rejects points whose depth differs too much from their neighbors. Higher = cleaner silhouettes but may clip thin objects.

Common Mistakes

Wire depth into Output.z: The scene compresses into a flat 0–1 slab. depth is nearness (0-1), Output.z expects signed geometry (−n to +n). Always wire z → Output.z, not depth.

Wire z into Palette: Colors become meaningless because z is signed and unbounded (−7.5 at 4m). Palette expects 0–1. Always wire depth → Palette.t.

NEAR too low on TrueDepth: The front camera reads down to ~10cm. At NEAR=0.05, close surfaces plateau into a flat cap because depth resolution saturates. Set NEAR ≥ 0.1 for TrueDepth.

Pinout

GRID GPU node. The pin-screen display — this is where your cloud becomes visible. XY snaps every point back to its fixed home grid position. Only the Z push (real metric depth) remains, creating a physical pin-screen toy relief.

How It Works

Every point in the cloud has a "home" — a fixed XY position on the grid determined by the Domain topology. The Depth node produces position offsets that displace points from home. Pinout lets you control how much of that offset reaches Output:

MORPH = 0: The classic pin-screen. XY is locked to the grid — points cannot move sideways. Only Z extends. This is the clean, architectural look.

MORPH = 1: The raw cloud. Points keep their full 3D offsets from Depth. The cloud floats freely in space. Use this for immersive, volumetric looks.

MORPH between 0 and 1: A blend. The cloud hovers near the grid, partially constrained. Wire an LFO into morph and the cloud breathes between rigid and free.

GAIN: Scales the Z push. At 1, the metric depth passes through unchanged. At 2, depth is doubled — more dramatic relief. At 0, the cloud goes completely flat against the wall.

Parameters

ParamRangeDefaultWhat It Does
morph0 – 100 = grid-locked pin-screen. 1 = free-floating cloud. Blend between for hybrid looks.
gain0 – 21Z push multiplier. 0 = flat wall. 1 = true metric depth. 2 = exaggerated relief.

Size

SHAPE GPU node. Controls how large each point appears on screen. Final size = base × input, clamped to [min, max]. Size 0 hides the pin entirely — use this for visibility masking.

How It Works

Size is the final multiplier before the vertex shader draws each point. The formula is simple: output = clamp(base × input, min, max). But the consequences are rich:

Constant size: Leave the size input unwired. Every pin uses the BASE value. A uniform cloud.

Depth-driven size: Wire Depth.depth into the size input. Near points are big, far points are small — natural perspective scaling. But depth alone gives a limited range; use a Remap node between them to set the exact size range you want (e.g., Remap outMin:0.1 outMax:3 → Size).

Audio-driven size: Wire Audio Volume through t-Remap into the size input. Points pulse with the music.

Masking: Wire a Hide node output, or a Region node mask directly into size. Pins outside the mask get size 0 → invisible. This is how you create spotlights, vignettes, and selective reveals.

Parameters

ParamRangeDefaultWhat It Does
base0 – 41Multiplier applied before clamping. At base=2, the same input produces twice the size.
min0 – 20Floor. Size never goes below this. At min=0.1, even "hidden" pins leave a faint speck.
max0.1 – 63Ceiling. Size never exceeds this. Prevents runaway sizes from loud audio or extreme depth.

Shockwave

MOVE GPU node. Expanding rings fired by triggers. Up to 4 waves alive simultaneously — each ring travels outward from its birth point, pushing points as it passes. Pure math — zero per-pin state.

How It Works

When the fire port receives a trigger pulse (1 on a frame), a new wave is born at the current center position. The wave expands outward over time: at each pin, the shader computes the distance from the wave center and checks if the ring front has reached that pin yet. If the pin is inside the active ring band, it gets displaced. The displacement is strongest at the ring center and falls off toward the edges of the ring width.

Up to 4 waves can be alive at once. When a 5th fires, the oldest wave is replaced. Each wave decays naturally — DAMPING controls how fast the ring amplitude fades as it travels.

Common pattern: Beat Detect → Shockwave.fire. Shockwave.pulse → Depth Drive.gain (exposed). Every kick drum sends a ripple through the cloud.

Multiple triggers: Wire Gestures.fist AND Gestures.palm both into Shockwave.fire (via t-Or). Different gestures fire waves from different screen positions.

Parameters

ParamRangeDefaultWhat It Does
speed0 – 41.2How fast the ring expands outward. At 1.2, a wave crosses the frame in about 0.4 seconds.
width0 – 0.60.12Ring thickness. At 0.12, the displacement band is about 12% of the frame width. Thinner = sharper shock. Wider = softer push.
damping0 – 61.2How fast the wave amplitude decays. Low damping = rings travel far before fading. High damping = quick localized pulse.

Palette

COLOR GPU node. Maps a 0–1 value through a color lookup table. The primary colorization node — without it, all points render white.

How It Works

The Palette node takes any fieldFloat input (typically Depth.depth, which is 0–1 nearness) and uses it as an index into a color gradient. At t=0 you get the leftmost color of the gradient. At t=1 you get the rightmost. The SHIFT parameter offsets this lookup — at shift=0.5, what was red becomes green (wrapping around the gradient).

Common pattern: Depth.depth → Palette.t → Output.color. Near objects (depth=1) get hot colors. Far objects (depth=0) get cold colors. This is the classic thermal LiDAR look.

Beyond depth: Wire anything 0–1 into Palette.t. Noise.out for organic color fields. Sine.out for pulsing color cycles. Remap.out to compress or expand the color range. Audio Volume through t-Remap for sound-reactive palette shifts.

Noise

SIGNAL GPU node. Animated 3D noise field computed per-pin on the GPU. Six noise types with fractal octave layering. Produces both a scalar output (0–1) and an optional color output through a palette map. The most versatile modulation source after Depth.

How It Works

Noise samples a coherent 3D field using each pin's grid position plus time. Because the field is continuous, neighboring pins get similar values — creating organic, flowing patterns rather than random static. The field "moves" along one axis driven by time, producing smooth animation without any per-pin state.

Fractal layering: HARMONICS controls how many octaves are summed. Each octave is higher frequency (by HARMONICSPREAD) and lower amplitude (by HARMONICGAIN). ROUGHNESS blends toward more high-frequency energy. The result: from smooth clouds (1 harmonic) to detailed turbulence (6 harmonics, high roughness).

typeperlin / simplex / random / sparse / hermite / harmonicperlinPerlin: Classic smooth gradient noise. Simplex: Less directional artifacts. Random: White noise per pin. Sparse: Clustered pockets of activity. Hermite: Smoother interpolation. Harmonic: Sum-of-sines — periodic and musical.
period0.02 – 40.5Feature size. 0.02 = fine grain like sand. 4 = broad sweeping waves across the whole frame.
harmonics1 – 63Octave count. 1 = smooth and simple. 6 = rich detailed turbulence (more GPU cost).
